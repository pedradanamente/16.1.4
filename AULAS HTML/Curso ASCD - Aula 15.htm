 PROTOCOLO FTP


Na aula anterior vimos como conectar e logar em um servidor utilizando o cliente FTP existente no Windows FTP.exe, porem os comandos utilizados foram comandos do cliente e não propriamente do protocolo. Nesta aula vamos ver como é a comunicação entre cliente e servidor FTP (Protocolo FTP).

Uma conexão FTP é composta por dois canais de comunicação:
1-) (Protocol Interpreter) Canal responsável pelos comandos entre cliente e servidor
2-) (Data Transfer Process) Canal responsável pelos dados transmitidos entre cliente e servidor.

Exemplo:



O “PI Servidor” é fica aguardando os comandos vindos do “PI Cliente”, depois de recebidos os comandos ele interpreta e pilota o DTP servidor.


COMANDOS FTP

Os comandos FTP permitem precisar:

  • A porta utilizada
  • O modo de transferência dos dados
  • A estrutura dos dados
  • A natureza da ação a efetuar (Retrieve, List, Store,…)


Tipos de comandos FTP:
  • Os comandos de controlo de acesso
  • Os comandos de parametrização de transferência
  • Os comandos de serviço FTP

LISTA DE COMANDOS

"Não se preocupe em decorar todos os comandos do protocolo"

COMANDOS DE CONTROLE DE ACESSO
USER Cadeia de caracteres que permitem identificar o utilizador. A identificação do utilizador é necessária para estabelecer uma comunicação no canal de dados
PASS Cadeia de caracteres que especificam a palavra-passe do utilizador. Este comando deve ser imediatamente precedido do comando USER. Cabe ao cliente esconder a afixação deste comando por razões de segurança
ACCT Cadeia de caracteres que representam a conta (account) do utilizador. Este comando não é geralmente necessário. Aquando da resposta à aceitação da palavra-passe, se a resposta for 230 esta fase não for necessária; se a resposta for 332, é.
CWD Change Working Directory : este comando permite alterar o directório corrente. Este comando necessita o caminho de acesso ao directório a atingir como argumento
CDUP Change to Parent Directory :este comando permite subir ao directório parente. Foi introduzido para remediar os problemas de nomeação de directório parente de acordo com o sistema (geralmente “. ”)
SMNT Structure Mount :
REIN Reinitialize :
QUIT Comando permitindo terminar a sessão corrente. O servidor espera até terminar a transferência corrente se for caso disso, seguidamente fornece uma resposta antes de fechar a conexão

COMANDOS DES PARÂMETROS DE TRANSFERÊNCIA
PORT Cadeia de caracteres que permitem precisar o número de porta a utilizar
PASV Comando permitindo indicar ao servidor DTP para esperar uma conexão numa porta específica escolhida aleatoriamente entre as portas disponíveis. A resposta a este comando é o endereço IP da máquina e a porta.
TYPE Este comando permite precisar o tipo de formato em que os dados serão enviados
STRU Carácter Telnet precisando a estrutura do ficheiro (F para File, R para Record, P para Page)
MODE Carácter Telnet precisando o modo de transferência dos dados (S para Stream, B para Block, C para Compressed)

COMANDOS DE SERVIÇO FTP
RETR Este comando (RETRIEVE) pede ao servidor DTP uma cópia do ficheiro cujo caminho de acesso passou em parâmetro.
STOR Este comando (store) pede ao servidor DTP que aceite os dados enviados sobre o canal de dados e que os armazene no ficheiro que leva o nome passado em parâmetro. Se o ficheiro não existir, o servidor cria-o, se não esmaga-o
STOU Este comando é idêntico ao precedente, mas pede ao servidor que crie um ficheiro cujo nome é único. O nome do ficheiro é devolvido na resposta
APPE Graças a este comando (append) os dados enviados são concatenados no ficheiro que leva o nome passado em parâmetro , se ele já existir; no caso contrário, é criado.
ALLO Este comando (allocate) pede ao servidor que preveja um espaço de armazenamento suficiente para conter o ficheiro cujo nome passou em argumento.
REST Este comando (restart) permite retomar uma transferência onde tinha parado. Para isso, este comando envia em parâmetro o indicador que representa a posição no ficheiro onde a transferência tinha sido interrompida. Este comando deve ser imediatamente seguido de um comando de transferência.
RNFR Este comando (rename from) permite rebaptizar um ficheiro. Indica em parâmetro o nome do ficheiro a rebaptizar e deve ser imediatamente seguido do comando RNTO
RNTO Este comando (rename to) permite rebaptizar um ficheiro. Indica em parâmetro o nome do ficheiro a rebaptizar e deve ser imediatamente precedido pelo comando RNFR
ABOR Este comando (abort) diz ao servidor DTP para abandonar todas as transferências associadas ao comando precedente. Se nenhuma conexão de dados estiver aberta, o servidor DTP não faz nada, se não fecha-a. Em contrapartida, o canal de controlo permanece aberto.
DELE Este comando (delete) permite suprimir o ficheiro cujo nome passou em parâmetro. Este comando é irremediável, só uma confirmação a nível do cliente pode ser feita.
RMD Este comando (remove directory) permite suprimir um directório. Indica em parâmetro o nome do directório a suprimir
MKD Este comando (make directory) permite criar um directório. Indica em parâmetro o nome do directório a criar
PWD Este comando (print working directory) permite reenviar o caminho completo do directório corrente
LIST Este comando permite reenviar a lista dos ficheiros e directórios presentes no directório corrente. Esta lista é enviada para o DTP passivo. É possível passar em parâmetro deste comando um nome de directório, o servidor DTP enviará a lista dos ficheiros no directório passado em parâmetro
NLST Este comando (name liste) permite enviar a lista dos ficheiros e directórios no directório corrente
SITE Este comando (site parameters) permite ao servidor propor serviços específicos, não definidos no protocolo FTP
SYST Este comando (system) permite enviar informações para o servidor distante
STAT Este comando (status) permite emitir o estado do servidor, por exemplo para conhecer a progressão de uma transferência corrente. Este comando aceita em argumento um caminho de acesso, reenvia então as mesmas informações que LIST mas sobre o canal de controle
HELP Este comando permite conhecer o conjunto das encomendas compreendidas pelo servidor. As informações são voltadas sobre o canal de controlo
NOOP Este comando (no operations) serve unicamente para obter um comando OK do servidor. Pode servir unicamente para não ser desligada após um tempo de inactividade demasiado longo


 


RESPOSTAS FTP

Andre, a cada requisição enviada pelo cliente o servidor responde de forma à garantir um sincronização.

As Respostas são constituídas por três números.

Os códigos de resposta são constituídos por 3 números (xyz), cujos significados são os seguintes:

    * O primeiro número indica o estatuto da resposta (sucesso ou falha)
    * O segundo número indica a que é que a resposta se refere
    * O terceiro número dá um significado mais específico (relativo a cada segundo número)

PRIMEIRO NUMERO
1yz Resposta preliminar positiva A ação pedida está curso, uma segunda resposta deve ser obtida antes de enviar um segundo comando
2yz Resposta positiva de realização A ação pedida foi realizada, um novo comando pode ser enviado
3yz Resposta intermédia positiva A ação pedida está temporariamente suspensa. Informações suplementares são esperadas por parte do cliente
4yz Resposta negativa de realização A ação pedida não teve lugar porque o comando temporariamente não foi aceite. O cliente deve experimentar de novo mais tarde
5yz Resposta negativa permanente A ação pedida não teve exito porque o comando não foi aceito. O cliente deve formular um pedido diferente


SEGUNDO NUMERO
x0z Sintaxe A acção possui um erro de sintaxe, ou trata-se de um comando não percebido pelo servidor
x1z Informação Trata-se de uma resposta que devolve informações (por exemplo, para uma resposta a um comando STAT)
x2z Conexões A resposta refere-se ao canal de dados
x3z Autenticação e contas A resposta refere-se à ligação (USER/PASS) ou ao pedido de mudança de conta (CPT)
x4z Não utilizado pelo protocolo FTP
x5z Sistema de arquivos A resposta refere-se ao sistema de arquivos distante

Exemplos de mensagens:

220 = A ação pedida foi realizada, um novo comando pode ser enviado, A resposta refere-se ao canal de dados

150 = A ação pedida está curso, uma segunda resposta deve ser obtida antes de enviar um segundo comando, A resposta refere-se ao sistema de arquivos distante.

ATENÇÃO !!
Veja que no cliente temos os nomes nos arquivos e no túnel ele não apareceu. Mas se é um túnel por onde os dados passaram?

A resposta está no começo da aula, a comunicação é feita através de dois canais e estamos “tunelando” apenas o PI canal de comandos e ano da dados, os nomes dos arquivos são dados enviados pelo DTP do servidor em outro canal (canal de transferência de dados).

 Em que porta os dados transmitidos?

 Observe o comando PORT realizado pelo cliente.
 PORT 127,0,0,1,5,242

Os quatros primeiros dígitos representam o IP do servidor, em nosso caso 127.0.0.1
Os dois últimos dígitos representam a porta utilizada, mas existe um calculo para determinarmos esta porta.


Calculo do numero da porta do comando PORT
Pegue o primeiro numero e multiplique por 256 e some ao segundo.

Em nosso caso 5 e 242
(5 X 256) + 242 = 1522

 Portanto:
 PORT 127,0,0,1,5,242 significa que o dado se encontra no IP 127.0.0.1 na porta 1522

 LOGINS ANÔNIMOS

 Andre, alguns servidores permitem login de forma anônima, isto torna público os arquivos encontrados no servidor, você não precisa de permissão para ter acesso aos arquivos. Em alguns casos são intencionais mas em outros casos se trata de "falta de atenção" do administrador do servidor.

Como logar em um FTP com arquivos públicos?
Basta usar o seguinte login:
Usuario: anonymous
Senha: IEUser@

Exemplo de FTP com arquivos públicos:
ftp.microsoft.com



Como saber se o servidor aceita login anonimo?

Você pode tentar o logim manualmente ou utilizar o SCAN FTP do Insecuritynet 3.X
iniciar -> Scan e monitoramento -> Scan FTP



Observe:  Anonimous Login..:+OK


ANTIGOS EXPLOITERS UTILIZANDO O COMANDO QUOTE

O comando quote presente no cliente "ftp.exe" permite que você envie comandos diretamente ao servidor, sem ser interpretados pelo cliente.
 Antigamente algums servidores possuem falhas que permitiam acesso de usuário apenas injetando comandos de logins diretamente ao servidor, atualmente é muito raro você encontrar algum servidor vulnerável.

 Segue abaixo uma lista de logins:

Exploit 1
quote user ftp
quote cwd ~root
quote pass ftp

Exploit 2
quote user ftp
quote cwd ~sys
quote pass ftp

Exploit 3
quote user ftp
quote cwd ~sysop
quote pass ftp

O comado cwd é utilizado originalmente para abrir um diretorio, netse exemplo "cwd ~root" se voce abrir o diretorio root você terá acesso total ao servidor.



PRINCIPAL VULNERABILIDADE EM SERVIDORES FTP

Alguns servidores ftp deixam arquivos.ini expostos, estae arquivos podem ser facilmente ser anexado ao google.
 Você poderá facilmente encontrar várias logins e senhas apenas com uma busca no "google".

 Pesquise por:
 filetype:ini ws_ftp pwd


VULNERABILIDADES DO PROTOCOLO FTP

 Sem dúvida o grande problema do protocolo FTP é a ausência de criptografia, as sessôes de logins trafegam livremente na Internet e sua senhas poderá ser interceptada. Um exemplo foi a nossa experiência que fizemos agora pouco, caso o túnel estivese em um computador remoto a senha poderia ser visualizada.
 Alem do túnel os logins podem ser capturados em redes privadas com SNIFFERs.
