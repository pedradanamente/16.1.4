 PROTOCOLO HTTP
 

O objectivo do protocolo HTTP é permitir uma transferência de ficheiros (essencialmente no formato HTML) localizados graças a uma cadeia de caracteres chamada URL entre um navegador (o cliente) e um servidor Web. Resumindo é o protocolo utilizado para você poder visualizar um website.

URL = Endereço Virtual exemplo (http://www.insecuritynet.com.br)

Funcionamento do protocolo HTTP

1-) O cliente (navegador) faz o pedido
2-) O servidor responde com os dados



 PORTA PADRÃO DO PROTOCOLO HTTP:
 Porta 80


Pedido HTTP

Andre, o pedido HTTP são os comandos enviados pelo navegador e interpretado pelo servidor, o pedido deve conter:

Linha de pedido
                 O método
                 O URL (endereço)
                 Versão do protocolo utilizada pelo cliente (navegador)

Os campos de cabeçalho do pedido         
             Informações sobre o navegador, sistema operacional, etc.

O corpo do pedido
              Conjunto de linhas opcionais que permitem envio de dados
Exemplo de um pedido HTTP
GET/ HTTP/1.1 
Accept : text/html 
If-Modified-Since : Saturday, 15-January-2010 12:25:51 GMT 
User-Agent : Mozilla/4.0 (compatible; MSIE 5.0; Windows 98)


COMANDOS HTTP
GET Pedido do recurso situado na URL especificada
HEAD Pedido do cabeçalho do recurso situado na URL especificada
POST Envio de dados ao programa situado na URL especificada
PUT Envio de dados à URL especificada
DELETE Supressão do recurso situado na URL especificada


Rublicas
Accept  Tipo de conteúdo aceito pelo motor de pesquisa (por exemplo text/HTML).
Accept-Charset Jogo de caracteres esperado pelo motor de pesquisa
Accept-Encoding Codificação de dados aceito pelo motor de pesquisa
Accept-Language Linguagem esperada pelo motor de pesquisa.
Authorization Identificação do motor de pesquisa junto do servidor
Content-Encoding Tipo de codificação do corpo do pedido
Content-Language  Tipo de linguagem do corpo do pedido
Content-Length Comprimento do corpo do pedido
Content-Type Tipo de conteúdo do corpo do pedido (por exemplo text/HTML).
Date Data de início de transferência dos dados
Forwarded Utilizado pelas máquinas intermédias entre o motor de pesquisa e o servidor
From Permite especificar o e-mail do cliente
From Permite especificar que o documento deve ser enviado se tiver sido alterado a partir de uma certa data
Link Relação entre duas URL
Orig-URL URL de origem do pedido
Referer URL da ligação a partir da qual o pedido foi efectuado
User-Agent Cadeia dando informações sobre o cliente, como o nome e a versão do navegador, do sistema de exploração





RESPOSTA HTTP

Andre, as respostas HTTP são os comandos e dados enviados de servidor para o navegador.
A resposta HTTP deve conter:

Uma linha de estatuto
Esta linha determina:
  • A versão do protocolo utilizado
  • O código de estatuto
  • A significado do código

Os campos de rubrica da resposta
  • Permite o envio de dados do servidor

Corpo da resposta
  • Dados pedidos


Exemplo de resposta HTTP
HTTP/1.0 200 OK 
Date : Sat, 15 Jan 2000 14:37:12 GMT 
Server : Microsoft-IIS/2.0 
Content-Type : text/HTML 
Content-Length : 1245 
Last-Modified : Fri, 14 Jan 2010 09:28:13 GMT

Corpo da resposta
Content-Encoding Tipo de codificação do corpo da resposta
Content-Language  Tipo de linguagem do corpo da resposta
Content-Length Comprimento do corpo da resposta
Content-Type Tipo de conteúdo do corpo da resposta (por exemplo text/HTML).
Date Data de início de transferência dos dados
Expires Data limite de consumo dos dados
Forwarded Utilizado pelas máquinas intermédias entre o motor de pesquisa e o servidor
Location Redireccionamento para uma nova URL associada ao documento
Server Características do servidor que envia a resposta




Códigos de resposta:
O código de resposta é constituído por três números: o primeiro indica a classe de estatuto e seguintes a natureza  do erro.
10x Mensagem de informação Estes códigos não são utilizados na versão 1.0 do protocolo
20x Sucesso Estes códigos indicam o bom desenrolar da transacção
200 OK O pedido foi realizado corretamente
201 CREATED

Segue um comando POST indica o sucesso, o corpo do resto do documento deve indicar a URL onde o documento recentemente criado deveria encontrar-se.
202 ACCEPTED O pedido foi aceito, mas o procedimento seguinte não foi realizado
203 PARTIAL INFORMATION Quando este código é recebido em resposta a um comando GET, isto indica que a resposta não está completa.
204 NO RESPONSE O servidor recebeu o pedido mas não há informação a devolver
205 RESET CONTENT O servidor indica ao navegador para suprimir o conteúdo dos campos de um formulário
206 PARTIAL CONTENT Trata-se de uma resposta a um pedido que comporta a rubrica range. O servidor deve indicar a rubrica content-range
30x Redireção Estes códigos indicam que o recurso já não está no lugar indicado
301 MOVED Os dados pedidos foram transferidos para um novo endereço
302 FOUND Os dados pedidos são de uma nova URL, contudo talvez tenham sido deslocados desde então
303 METHOD Isto implica que o cliente deve tentar um novo endereço, tentando preferivelmente um outro método além do GET
304 NOT MODIFIED Se o cliente efectuar um comando GET condicional (perguntando se o documento foi alterado desde a última vez) e se o documento não tiver sido alterado, devolve este código.
40x Erro devido ao cliente Estes códigos indicam que o pedido está incorrecto
400 BAD REQUEST A sintaxe do pedido está mal formulada ou é impossível de satisfazer
401 UNAUTHORIZED O parâmetro da mensagem dá as especificações das formas de autorização aceitáveis. O cliente deve reformular o seu pedido com os bons dados de autorização
402 PAYMENT REQUIRED O cliente deve reformular o seu pedido com os dados de pagamento
403 FORBIDDEN O acesso ao recurso é simplesmente proibido
404 NOT FOUND O servidor não encontrou nada no endereço indicado.
50x Erro devido ao servidor Estes códigos indicam que houve um erro interno do servidor
500 INTERNAL ERROR O servidor encontrou uma condição inesperada que o impediu de satisfazer o pedido
501 NOT IMPLEMENTED O servidor não suporta o serviço pedido
502 BAD GATEWAY O servidor recebeu uma resposta inválida por parte do servidor que tentava aceder agindo como uma ponte estreita ou um proxy
503 SERVICE UNAVAILABLE O servidor não pode responder-lhe no momento presente, porque o tráfego é demasiado denso
504 GATEWAY TIMEOUT A resposta do servidor foi demasiado longa no que diz respeito ao tempo durante o qual a ponte estreita estava preparada para o esperar







Criando um servidor HTTP em seu computador.

O Insecuritynet 3.X possui um simples HTTP server para você hospedar pequenas paginas em HTML em seu computador.
 
 Vamos abrir o servidor, entre em:
 Iniciar -> servidores -> HTTP
 
 Entre em "Abrir Diretório"
 
 

  Automaticamente vai abrir uma pasta.
 
  Copie seu site para esta Pasta.
  Importante: A pagina inicial deve conter o nome: index.html
   
  Caso você não possua nenhum site pronto, clique aqui e baixe este exemplo. (Senha:1234)
   
   

   Agora vamos Iniciar o Servidor (Colocar a pagina no ar)

   No servidor entre em: Iniciar servidor

    Automaticamente o servidor será aberto em outra janela (não feche esta janela, pois é o servidor ativo)
   
   

     Agora vá ate seu navegador e Digite o Endereço de seu site:

     

 Observe que o servidor registra as as arquivos requeridos na visualização do site:

 

  Para fechar o servidor basta fechar esta janela do Prompt "HTTP SERVER".

  (Bloqueio da porta 80) Sobre a porta utilizada.

   A porta padrão para HTTP é a porta 80, porem na experiência utilizamos a porta 8085.
   Por motivos de segurança e banda de tráfego, vários provedores de Internet bloqueiam a porta 80, sendo assim voce nao pode hospedar um servidor HTTP em seu computador e disponibilizar na Internet.
   Utilizando outra porta (por exemplo: 8085) seu provedor não irá bloquear e você poderá colocar sua pagina no ar.
   Importante: Para sua pagina estar no ar, você deve se certificar que seu IP não é NAT, ou seja, válido na Internet.  É o mesmo processo de se hospedar um servidor de jogo.
   Este servidor do Insecuritynet é apenas didático, sem maiores funções, você poderá apenas hospedar seu site em formato HTML e arquivos para downloads. Noa tem suporte para PHP, ASP, etc.
   Caso voce deixe seu servidor online na Internet você poderá visualizar os LOGs de conexões, entre em:
   ...Meus documentos\Insecuritynet\LOGS\http_server.log

COMPRESSÃO HTTP

PORQUE CODIFICAR em GZIP?
Ao contrario do que parece não é por motivos de segurança, mas sim de desempenho.
O GZIP nada mais é do que uma compressão, é como se comptáctassemos um arquivo no Winrar ou Winzip.
O tamanho de arquivo de texto fica muito menor, um código HTML nada mais é do que uma sequência de comandos em modo texto, portanto sua compressão irá reduzir muito seu tamanho.
 Para que esta compressão seja possível, o servidor deve compactar o código e o navegador deve "descompactar". Isto não envolve segurança porque os dados podem ser facilmente decodificados, mas proporciona uma enorme economia de tempo e largura de banda na web, a economia media varia entre 60 1.677354e-307 85
